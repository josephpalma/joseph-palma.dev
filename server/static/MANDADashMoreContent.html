Project Challenge

When crisis arises, Marine Corps Commanders must quickly orient, observe and assess the situation. Commander’s employ logisticians to inform the course of action by providing estimates on air and sea port attributes and assets. Estimates needed to make a safe determination can take planners days per key terrain location. Time spent manually aggregating hundreds of data points takes time away from Commanders - which can result in catastrophic repercussions for our nation, friendly forces, partners and allies.

Project Solution

The “Maritime and Aviation Dashboard” or Manda Dashboard allows logisticians to conduct estimates much faster by providing quick data cards with port, airfield, and throughput capacities on specific locations. The Dashboard achieved the mission statement by aggregating hard-to-access information, accessible in a user friendly dashboard, improving the agility of the logistical team and enabling faster decisions for Commanders.

Bureaucratic Web Architecture

Several military projects over the past century have concentrated on developing speed and scalability for classified communication, in fact, the Internet itself was the outcome of a

military project.

However, modern protocols often create roadblocks that can stall logistical teams.

Client Pain Points

Existing logistical tools run on a secure private network, known as the Secure Internet Protocol Router Network (SIPRNet). Reliance on Controlled Unclassified Information (CUI) segregates publicly available knowledge from many SIPRNet applications. Complicated network permissions make it difficult for non-logisticians in the planning process to access the essential data they provide.

Key Findings

  • >

    App must operate on the civilian internet. No authorization, no problem.

  • >

    Data set must be:

    • >

      Open source and non-CUI compliant for added transportability and security.

    • >

      Easily integrable into SIPRNet programs, a route for public information.

Understand, Ideate and Iterate

Our sponsors elected for Tableau and Python, because of the ubiquity of Python in SIPRNet applications. However, when they learned about the license costs Tableau was scratched. Instead, we worked with them to create a tech stack that would suit their needs and help us achieve the mission. There was only one constraint: zero cash overhead.

Design v1

Now it was time to start building. We approached our first system design looking to hit a homerun. However, we quickly found out we were over-scoped. Design v1 contained features like route planning, obstacle detection, and a reachability function to search locations based on vehicle characteristics. Unfortunately, most of these features were completely reliant on CUI data! So, we abandoned this design and started new.

Design v2

Our second system design stripped back all the bells and whistles, and instead focused on scraping and displaying important data. We worked with our sponsors to determine an initial set of data points – while leaving room for expansion. We were now at a good place to mock up a UI design. A common trend was noticed among logistical presentations: displaying stats in a baseball card like fashion. This was our inspiration for a card-based UI.

Niche Data Requirements

To start, our sponsors compiled a list of ~100 sources, however most were locked up behind the SIPRNet permissions wall. So we set out to find our own alternate, reliable sources - a task we ended up investing about ⅓ of our time into, because as it turns out, searching “mooring points at Port Aitape, Papua New Guinea” doesn’t turn up any useful information.

Most value ended up coming from Searates, World Port Index, Pub150 Index, and Maxar. Altogether we succeeded in sourcing 94% of the original data points

from alternative sources - much higher than we expected.

Open Source Solution

Logisticians need to be able to conduct estimates much faster. By sourcing only publicly available information, we removed the need for authorization, reducing wait time considerably.

Data Flow

With the goal of data autonomy and flexibility we designed a backend system that provides information in universal formats.

Data is scraped into .csv’s where it is distributed to SIPRNet, and our MongoDB instance for use on the front-end. Everytime the scraper runs, backups are written to a separate DB, and each dataset is loaded with the latest information.

Plan of Attack

Assemble Data Structures

We worked with our sponsors to define a structure from our data points.

Build Scraping Infrastructure

Some of our sources had protections against web crawlers, we were able to avoid them by implementing scraper timeouts. While this did extend the time to scrape - it also allowed for a much more thorough dataset.

Efficient Presentation of Data

We were able to pull non-CUI publicly available information on certain vehicles, like C-130 and C-17 aircrafts, as well as first-hand descriptions. Locations without first-hand descriptions used satellite images with embedded port and airfield asset descriptions.

Dynamic Error Logging

Many things could go wrong when scraping the web. Broken links, blocked connections or changes in formatting can all result in incomplete data. We want all users to have insight into error messages, and suggestions on how to fix them, so appropriate measures can be taken in the case of failure. To do this we implemented a logging system, which users can interact with from the front-end.
